# Projects Context

## Glossary

| Term | Definition | Not this |
|------|-----------|----------|
| **Appointment** | A scheduled moment when artist and client are together. Has a type (tattoo, consultation, touch-up, quick) and a lifecycle: Scheduled → In Progress → Finalized → (optionally) Paid. "Finalized" means "this happened and is complete" — for tattoo types, Materials are sealed; for others, it's just a timestamp. Paid is optional (free consultations, complimentary touch-ups can sit in Finalized forever). Traceability records are created only for tattoo-type Appointments. Cross-profile blocking: client self-booking hard-blocks occupied slots; artist can manually create overlapping Appointments (soft warning). Always has a `workplaceId` (determines shared inventory shown during Material recording, location on calendar). | Not "Session" (ambiguous with auth/agent sessions, and "In Session" is awkward as a state name). |
| **Project** | The full lifecycle of a tattoo from submission to completion. Contains one or more Appointments. State machine: New → Reviewing → Quoted → Booked → In Progress → Completed (terminal: Declined, Cancelled). Declined carries metadata: `declinedBy` (artist or customer), `declinedAt`, optional `reason`. | |
| **Customer** | A person who receives tattoos. Identified by name + at least one contact (email or phone). Auto-created from project requests. Scoped to a single Artist Profile. In code: `Customer`. In UI copy: "Client" (what artists actually say). | Not "Client" in code (collides with `octattoo_client` package, HTTP clients, Dart's `HttpClient`). |
| **GDPR Erasure** | Anonymization of a Customer's personal data (name, email, phone, social handle, Consent Package PDFs, health data) while preserving traceability. After erasure: traceability records retain an opaque `anonymized_customer_id` (stable, so "these 3 appointments were the same person" is knowable without identifying who), Appointment dates, Materials used, and a boolean "consent was obtained" + signature date. All personal data is gone. Satisfies both GDPR Art. 17 and EN 17169:2019. |
| **Quick Appointment** | An Appointment created without a parent Project — for walk-ins, touch-ups, convention flash. Same lifecycle and traceability as project-linked Appointments. Can be immediate (starts In Progress now) or scheduled for a future date/time (starts in Scheduled state). |
| **Consent Package** | A paired unit: health questionnaire + consent form + signature. Standalone entity — optionally linked to an Appointment, but can exist independently (e.g., artist uses the app only for consent management). Always belongs to a Customer + Artist Profile. Immutable PDF generated server-side at moment of signature. | |
